## 0. 핵심 개념 — Realtime의 3가지 축

Supabase Realtime은 목적이 다른 3가지 메커니즘을 제공한다. 이 프로젝트에서는 셋을 용도별로 나눠서 사용한다.

| 방식 | 특징 | 이 프로젝트에서의 용도 |
|---|---|---|
| **Broadcast** | DB를 거치지 않는 임시 신호, 저장되지 않음, 매우 빠름 | 드래그 중 위치, 편집 중 표시(소프트 락) |
| **Postgres Changes** | DB에 INSERT/UPDATE/DELETE 발생 시 자동 브로드캐스트 | 객체 생성/수정 확정/삭제/잠금 등 저장 완료된 상태 |
| **Presence** | 채널에 접속 중인 클라이언트 목록을 실시간 추적 | 상단바에 표시할 "현재 접속자" |

---

## 1. 채널 구조

보드 1개당 채널 1개를 사용한다.

```js
const channel = supabase.channel(`board-${boardId}`, {
  config: { presence: { key: myUserId } }
})
```

## 2. 이벤트 종류별 설계

| 이벤트 | 방식 | 발생 시점 | 페이로드 예시 |
|---|---|---|---|
| 객체 드래그 중 위치 | Broadcast | 드래그 중 매 프레임(throttle 적용) | `{ objectId, x, y }` |
| 객체 생성/수정 확정/삭제/잠금 | Postgres Changes | DB write 완료 시 자동 발생 | 변경된 row 전체 |
| 객체를 지금 만지고 있음 표시(소프트 락) | Broadcast | 드래그/편집 시작·종료 시 | `{ objectId, userId, action: 'dragging' \| 'editing' }` |
| 현재 접속자 목록 | Presence | 입장/퇴장 시 자동 | `{ userId, nickname }` |

### 2-1. 드래그 중 위치 — Broadcast, DB 미기록
- 초당 여러 번 발생하므로 DB에 매번 쓰지 않는다.
- 기존 원칙("드래그 종료 시점에만 저장")을 그대로 유지한다.

### 2-2. 확정된 변경 — Postgres Changes 자동 처리
- 프론트가 별도로 브로드캐스트를 보낼 필요 없이, DB에 쓰는 순간 Supabase가 다른 접속자에게 자동으로 전파한다.
- DB 상태와 화면 상태가 항상 일치하도록 보장된다.

### 2-3. 소프트 락 (동시 편집 표시)
- guest 권한이 동등하여 누구나 어떤 객체든 만질 수 있으므로, 동시에 같은 객체를 조작할 가능성이 있다.
- DB 레벨에서 편집을 막지는 않되(과도한 제약), 다른 사용자 화면에 "◯◯님이 편집 중" 표시 + 해당 객체 테두리 흐림 처리로 충돌 가능성을 UX 차원에서 낮춘다.
- 흐름: 드래그/편집 시작 → Broadcast(`action:'dragging'`) 송신 → 종료 시 동일 채널로 종료 신호 송신

### 2-4. 상단바 참여자 아바타 — Presence 기반으로 전환

> **기존 설계 수정 사항**: 보드메인화면_UI설계서.md에서 상단바 아바타를 `board_member` 테이블 전체 목록으로 표시하도록 그렸으나, 이는 "권한을 부여받은 적 있는 사람 전체"이지 "현재 접속 중인 사람"이 아니다. 상단바는 **Presence 기반(현재 접속자만)**으로 수정한다.

- `board_member`는 권한 관리(누가 owner/guest/watcher인지)에 사용
- Presence는 "지금 이 화면을 보고 있는 사람"만 실시간 반영에 사용
- 두 정보를 조합해 아바타에 역할별 색상을 표시 (Presence로 접속자 파악 → 해당 user_id의 role은 board_member 조회로 확인)

## 3. 초기 진입 시 동기화 순서

```
1. 보드 페이지 진입
2. REST로 board_objects 전체 조회 (현재 확정 상태 스냅샷)
3. Realtime 채널 구독 시작 (postgres_changes + broadcast + presence)
4. 이후 변경은 실시간으로만 반영
```
스냅샷 조회(2번)를 구독 시작(3번)보다 먼저 수행해야, 구독 시작 전에 이미 존재하던 데이터를 누락하지 않는다.

## 4. 연결 끊김 → 재접속 처리

Supabase 클라이언트는 재연결을 자동 시도하지만, **끊겨 있던 동안 발생한 변경사항은 채널로 전달되지 않는다.** 재연결 성공 시점에 전체 스냅샷을 다시 조회해 보정한다.

```js
channel.on('system', {}, (status) => {
  if (status === 'reconnected') {
    refetchAllObjects() // 놓친 변경사항 보정
  }
})
```

## 5. 채널 정리(cleanup) 정책

### 5-1. 서버 측: 자동
구독자가 0명이 되면 Supabase Realtime 서버가 해당 채널(topic)을 자동으로 비활성화한다. 별도 조치가 필요 없다.

### 5-2. 클라이언트 측: 명시적 처리 필요
"사람이 나갔다"는 것을 서버가 정확히 인지하려면, 프론트에서 구독 해제를 명시적으로 호출해야 한다.

```js
useEffect(() => {
  const channel = supabase.channel(`board-${boardId}`)
  channel.subscribe()

  return () => {
    supabase.removeChannel(channel) // 페이지 이동 시 반드시 호출
  }
}, [boardId])
```

| 상황 | 정리 방식 |
|---|---|
| 페이지 이동 (SPA 라우팅) | `removeChannel()`을 컴포넌트 unmount 시 명시적으로 호출해야 함. 누락 시 연결이 남아있을 수 있음 |
| 탭 닫기 | 네트워크 연결 자체가 끊겨 자연 정리됨 (서버 인지까지 하트비트 타임아웃만큼 지연 발생 가능) |
| 마지막 접속자 퇴장 | 서버가 채널을 자동 비활성화, 별도 조치 불필요 |

## 6. 동시 편집 충돌 처리 방침

| 유형 | 처리 방식 |
|---|---|
| 위치 이동 | Last-Write-Wins (마지막 저장이 반영됨). 소프트 락 표시로 충돌 빈도 자체를 낮춤 |
| 텍스트 편집 | 한 명이 편집 시작 시 다른 사용자 화면엔 "편집 중" 표시 → 사실상 동시 타이핑을 UX로 방지 (DB 레벨 잠금은 아님) |

## 8. 소프트 락 UI 상세

### 8-1. 시각적 표현

| 요소 | 스타일 |
|---|---|
| 라벨 | 객체 상단에 "◯◯님이 편집 중" pill 형태, 활동 표시 점 아이콘 포함 |
| 테두리 | 점선 + 강조색으로 변경 |
| 객체 본문 | 반투명 처리 |
| 상호작용 | `pointer-events: none` 적용 — 다른 사용자는 이 시점에 클릭/드래그 불가 (DB 레벨 잠금은 아니며 UX 차원의 방지) |

본인이 편집 중인 객체에는 이 라벨을 표시하지 않는다 (본인 화면 기준 불필요한 정보).

### 8-2. 타이밍

| 상황 | 처리 |
|---|---|
| 드래그/편집 시작 | Broadcast로 `action:'dragging'` 즉시 송신 → 다른 사용자 화면에 라벨 즉시 표시 |
| 드래그/편집 종료 | 종료 Broadcast 송신 → 라벨 즉시 제거 |
| 텍스트 편집 중 | 500ms 디바운스 저장과 별개로, 편집 종료(blur) 신호가 와야 라벨이 사라짐 |

### 8-3. 엣지 케이스 — 종료 신호 유실 시 자동 해제

드래그/편집 중이던 사용자의 연결이 갑자기 끊기면(탭 강제 종료, 네트워크 단절) 종료 Broadcast를 보내지 못한 채 사라질 수 있다. 이 경우 다른 사용자 화면에는 해당 객체가 "누군가 계속 편집 중"인 상태로 영구히 잠긴 것처럼 보이는 문제가 발생한다.

**해결: 타임아웃 기반 자동 해제 + 하트비트**

드래그 중에는 위치 업데이트 자체가 생존 신호 역할을 하지만, **텍스트 편집 중에는 가만히 생각하며 타이핑을 멈추는 구간이 있어 위치 변화가 없다.** 이 경우 실제로는 편집 중인데도 신호가 없어 락이 오판으로 풀릴 수 있으므로, 편집 중에는 별도 하트비트를 둔다.

```
편집 시작 → 4초마다 하트비트 Broadcast 송신 (내용 변경 여부와 무관하게)
드래그 중 → 위치 업데이트 자체가 하트비트 역할 (기존과 동일)

"편집 중" 신호 수신 → 라벨 표시 + 타이머 시작(8초)
8초 내 "종료" 신호 또는 하트비트(4초 주기)가 다시 오지 않으면 → 자동으로 라벨 해제
```

### 8-4. 타임아웃 값: 8초 (확정)

하트비트 주기(4초)의 2배로 설정해, 정상적인 편집 중에는 타임아웃 전에 항상 최소 1회 이상 하트비트가 도착하도록 여유를 둔다.

| 값 | 특징 |
|---|---|
| 5초 | 유령 락 노출은 짧으나, 네트워크가 살짝만 느려도 오작동 가능성 있음 |
| **8초 (채택)** | 하트비트(4초) 대비 2배 여유로 균형적 |
| 10초 | 여유는 충분하나 유령 락 노출 시간이 상대적으로 김 |

## 9. Presence - Role 조합 로직 상세

### 9-1. 데이터가 두 곳으로 나뉘는 이유

| 정보 | 저장 위치 | 갱신 빈도 |
|---|---|---|
| 이 사람이 지금 접속해 있는가 | Presence (메모리, 실시간) | 매우 잦음 (입장/퇴장마다) |
| 이 사람의 역할이 owner/guest/watcher인가 | `board_member` 테이블 (DB) | 드묾 (owner가 권한 변경 시만) |

상단바 아바타 렌더링은 이 둘을 조합해서 만든다: "지금 접속 중인 사람 목록" × "각자의 역할".

### 9-2. 데이터 흐름

```
1. 보드 진입 시
   → board_member 전체를 REST로 조회하여 로컬에 role map 저장
      { userId1: 'owner', userId2: 'guest', userId3: 'watcher' }

2. Presence 채널 구독
   → channel.track({ userId, nickname })
   → presence sync 이벤트로 "현재 접속 중인 userId 목록" 수신

3. 렌더링 시 두 데이터 조합
   접속중 userId 목록.map(userId => ({
     role: roleMap[userId],
     nickname: presenceState[userId].nickname
   }))

4. owner가 권한 변경 시 (board_member UPDATE)
   → postgres_changes 이벤트로 감지 → 로컬 roleMap만 갱신
   → Presence 쪽은 갱신 불필요 (접속 여부는 변하지 않음)
```

### 9-3. Presence payload 구조

```js
channel.track({
  userId: myUserId,
  nickname: myNickname,       // 구글 닉네임 또는 익명 유저용 임시 닉네임
  avatarColor: myAvatarColor  // 개인 식별용 색상(선택)
})
```

**주의: `role`은 Presence payload에 포함하지 않는다.** Presence는 클라이언트가 스스로 보고하는 정보이므로, role을 여기 넣으면 악의적 사용자가 자신을 `owner`로 조작해 broadcast할 수 있다. **role은 반드시 DB(`board_member`)에서만 조회**해 신뢰성을 확보한다.

### 9-4. 재접속 처리
- Presence는 연결이 끊기면 하트비트 타임아웃 기반으로 자동으로 해당 유저를 목록에서 제거
- 재접속 시 `track()`을 다시 호출해 목록에 복귀
- roleMap은 "재연결 시 전체 스냅샷 재조회" 로직(4장)에 `board_member` 조회도 함께 포함

### 9-5. 구조 요약

```
        [board_member 테이블]              [Presence 채널]
              |  (드묾, DB)                    |  (잦음, 메모리)
              |  postgres_changes 구독          |  presence sync 이벤트
              v                                v
        roleMap 로컬 상태               접속중 userId 목록
              \___________________________/
                          |
                  화면에 아바타로 조합 렌더링
                  (역할별 색상 + 닉네임)
```
