## 1. 로그인 / 회원가입 화면

### 설계 전제
구글 OAuth만 지원하므로 **로그인과 회원가입은 별도 화면으로 분리하지 않는다.** 최초 접속 시 자동 가입 처리, 기존 계정이면 로그인으로 처리된다.

### 화면 구성 (`/`)

```
┌─────────────────────────────┐
│                               │
│         [보드 아이콘]          │
│  함께 만드는 온라인 화이트보드   │
│  메모와 이미지로 아이디어를      │
│  자유롭게 정리해보세요          │
│                               │
│     [G] 구글로 계속하기        │
│                               │
│  별도 회원가입 없이 바로 시작    │
│                               │
└─────────────────────────────┘
```

| 요소 | 설명 |
|---|---|
| 서비스 소개 | 제목 + 짧은 설명 문구 |
| 로그인 버튼 | "구글로 계속하기" 단일 버튼 |
| 보조 문구 | "별도 회원가입 없이 구글 계정으로 바로 시작해요" |

### 로그인 완료 후 이동
```
구글 로그인 완료 → /boards (내 보드 목록)로 자동 이동
```

### 예외: 초대 링크로 접속한 경우
`/invite/:token`으로 바로 들어온 사용자는 이 랜딩 화면을 거치지 않고, **Supabase Anonymous Auth로 자동 로그인 후 곧바로 보드에 watcher로 입장**한다. 구글 로그인 화면을 별도로 보여주지 않는다.

## 2. 계정 연동 흐름 (익명 유저 → 구글 계정)

### 배경
초대 링크로 들어온 익명(anonymous) 유저는 브라우저 세션에 의존하기 때문에, 캐시를 지우거나 다른 기기로 접속하면 권한과 기록이 유지되지 않는다. 이를 보완하기 위해 보드 내에서 구글 계정을 연동할 수 있는 진입점을 둔다.

### 진입 지점
보드 상단바의 **본인 아바타 클릭** (익명 유저는 닉네임 없이 아이콘만 표시됨)

### 화면 구성 (팝오버)

```
┌───────────────────────────┐
│ 익명으로 참여 중             │
│ 구글 계정을 연동하면          │
│ 브라우저를 바꿔도 내 권한과     │
│ 기록이 유지돼요               │
│                             │
│  [G] 구글 계정 연동하기       │
└───────────────────────────┘
```

| 요소 | 설명 |
|---|---|
| 상태 안내 | "익명으로 참여 중" 표시 |
| 연동 유도 문구 | 연동 시 얻는 이점을 한 줄로 설명 |
| 연동 버튼 | 구글 OAuth 진행 |

### 처리 흐름

```
"구글 계정 연동하기" 클릭
→ 구글 OAuth 팝업/리다이렉트
→ 인증 완료
→ 기존 익명 세션(uuid)에 구글 계정을 link
  (Supabase Auth의 identity linking 기능 사용 — 새 계정 생성이 아님)
→ board_member.user_id는 변경되지 않으므로 기존 권한(guest/watcher) 그대로 유지
→ board_objects.created_by로 연결된 기존 생성 기록도 그대로 유지
→ 아바타가 익명 아이콘 → 구글 프로필 사진/닉네임으로 즉시 갱신
```

**핵심 원칙**: 연동은 "새 계정 생성 후 데이터 이전"이 아니라 "기존 uuid에 인증 수단만 추가"하는 방식이다. 그래야 이미 만든 객체나 부여받은 권한이 끊기지 않는다.

### 연동 전/후 비교

| 항목 | 연동 전 (익명) | 연동 후 |
|---|---|---|
| 아바타 표시 | 아이콘만 (닉네임 없음) | 구글 프로필 사진 + 이름 |
| 브라우저 캐시 삭제 시 | 세션 유실 → 재접속 시 watcher부터 다시 시작 | 재로그인 시 동일 계정으로 복귀 |
| 권한 유지 | 같은 브라우저 세션 내에서만 유지 | 영구 유지 |
| 생성한 객체 기록 | 세션 유실 시 소유 정보 사실상 무의미해짐 | 계정 기준으로 영구 추적 가능 |

## 3. 확정 사항

### 3-1. 초대 링크 접속 시 세션 처리
별도 분기 없이 **세션 유무 기준**으로 통일 처리한다.

```
초대 링크 접속
→ 브라우저에 유효한 세션이 있는가?
   - 없음 → Anonymous Auth로 신규 세션 생성 → watcher로 board_member 등록
   - 있음 (구글 로그인 상태) → 해당 user_id 그대로 사용 → watcher로 board_member 등록
   - 있음 (다른 보드에서 만든 익명 세션) → 그대로 재사용
```
"이미 로그인된 사람이 접속하는 경우"를 별도 케이스로 다루지 않고, 세션 확인 로직 하나로 통일한다.

### 3-2. 계정 연동 중 오류 처리 (이미 연동된 구글 계정)

동일 구글 계정이 다른 세션에 이미 연동돼 있어 linking이 실패하는 경우, 로그인 전환 옵션을 제공한다.

```
연동 시도 → 에러 발생 (identity already exists)
→ 확인 모달: "이 구글 계정은 이미 다른 곳에서 사용 중이에요. 대신 이 계정으로 로그인할까요?"
→ [이 계정으로 로그인] → 현재 익명 세션 종료, 기존 구글 계정으로 재로그인
→ [취소] → 익명 상태 유지, 연동 취소
```

**트레이드오프**: "이 계정으로 로그인"을 선택하면 직전 익명 세션에서 생성한 객체의 `created_by`는 새로 전환된 계정과 연결되지 않고 과거 익명 uuid로 남는다. guest 권한이 소유 여부와 무관하게 동등하게 부여되므로 기능상 문제는 없으며, `created_by`는 기록 목적으로만 남는다.
