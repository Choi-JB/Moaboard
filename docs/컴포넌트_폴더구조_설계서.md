## 2. 설계 기준

### 2-1. `pages/` vs `components/` 분리
- `pages/`는 라우트에 대응되는 최상위 화면. 각 화면 설계서 문서와 1:1로 매칭된다.
  - LandingPage.tsx ↔ 로그인_계정연동_화면설계서.md
  - BoardListPage.tsx ↔ 보드목록_화면설계서.md
  - BoardPage.tsx ↔ 보드메인화면_UI설계서.md
  - InviteEntryPage.tsx / InviteErrorPage.tsx ↔ 초대링크접속_화면설계서.md
- `components/`는 화면을 구성하는 조각. 특정 화면 전용(`board/`, `boardList/`)과 범용(`ui/`)을 분리해 위치를 예측 가능하게 한다.

### 2-2. `stores/`를 역할별로 분리한 이유
실시간동기화_상세설계서.md에서 정의한 "board_member(느림, DB)"와 "presence(빠름, 메모리)"의 분리 원칙을 스토어 구조에 그대로 반영한다. 하나의 스토어에 몰아넣으면 갱신 빈도가 다른 데이터가 섞여 불필요한 리렌더링이 발생할 수 있다.

### 2-3. `hooks/`에 실시간 관련 로직을 모은 이유
채널 구독/해제, 소프트 락 타임아웃 등은 컴포넌트 생명주기와 얽힌 부수효과이므로 훅으로 분리해 재사용성과 가독성을 확보한다.

### 2-4. `lib/api/`와 RLS/API 명세서의 대응
RLS정책_API명세서.md에 정리된 `supabase.from(...)` 호출 코드가 위치하는 자리. 컴포넌트가 Supabase를 직접 호출하지 않고 이 레이어를 거치게 하여, 쿼리 로직 변경 시 컴포넌트 수정 없이 대응 가능하도록 한다.
