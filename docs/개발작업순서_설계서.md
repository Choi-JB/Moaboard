## 2. 순서 결정 근거

### 2-1. 실시간 동기화를 3주차로 미루는 이유
로컬 상태에서 객체 CRUD(생성/이동/삭제)가 먼저 안정적으로 동작해야, 그 위에 실시간 로직을 얹었을 때 문제가 생겼을 때 **로컬 로직 문제인지 동기화 문제인지 구분하기 쉽다.** 처음부터 둘을 동시에 만들면 디버깅 난이도가 크게 올라간다.

### 2-2. 권한 시스템(owner/guest/watcher 분기)을 뒤로 미루는 이유
초반에는 권한 구분 없이 "모든 기능이 다 되는 상태"로 먼저 완성하고, 이후 RLS 정책과 프론트 조건부 렌더링(권한별 UI 노출/차단)을 얹는 순서가 더 안전하다. 처음부터 권한별 분기를 넣으면 매 기능을 테스트할 때마다 여러 역할로 반복 확인해야 해서 개발 속도가 느려진다.

## 3. 주차별 완료 기준 (체크포인트)

| 주차 | 완료 기준 |
|---|---|
| 1주차 | 구글 로그인 → 보드 목록 조회 → 보드 생성까지 정상 동작 |
| 2주차 | 혼자 접속한 상태에서 객체 생성/드래그/삭제/새로고침 후에도 유지됨 |
| 3주차 | 두 개의 브라우저 창(또는 시크릿 창)으로 동시 접속 시 실시간 반영 확인 |
| 4주차 | 권한별 UI 제한, 설정/삭제, 초대 링크 흐름까지 전체 시나리오 정상 동작, 배포 완료 |

## 4. 참고 — 우선순위가 흔들릴 때 판단 기준
시간이 부족해질 경우, 아래 순서로 기능을 줄인다 (뒤로 갈수록 먼저 포기 가능).
```
1. 반응형 디테일 다듬기
2. 소프트 락 UI (없어도 기능은 동작함, UX만 저하)
3. 계정 연동(익명→구글) 흐름
4. 권한 세부 제어(owner만 잠금 등) — 단, 기본 owner/guest/watcher 구분 자체는 유지
```
핵심 기능(실시간 동기화 자체)은 마지막까지 유지한다.
